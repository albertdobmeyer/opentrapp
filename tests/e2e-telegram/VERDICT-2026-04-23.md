# Verdict — Phase 0 + direct probing (2026-04-23)

**Short answer: the perimeter holds. Not Swiss cheese, not an empty cell — on the host-reachability axis, it's "just right."** The one real finding was a credential leak in the proxy's own observability layer (self-inflicted, not an attacker path), caught during Phase 0 setup and fixed before the automated harness even ran. Details below.

## What was tested

This session was supposed to build a Telegram-driven test harness, then use it to probe the perimeter. The harness scaffold is complete but Phase 3 (Telethon login) needs a human to paste a one-time code — we only completed up to that line tonight. To produce **real signal** before morning, a parallel suite of direct-container probes ran (`tests/e2e-telegram/direct_probing/probe.sh`): 24 probes via `podman exec` against the four running containers, no LLM cost, no Telegram.

## Findings

### Finding #1 — Telegram bot token leaked in vault-proxy stdout + JSONL logs ✅ FIXED

**Category:** Credential leak into observability layer.
**Surface:** `vault-proxy` logged the full Telegram Bot API URL on every request. The URL path embeds the token: `https://api.telegram.org/bot<id>:<hash>/<method>`. Both the stdout stream (`podman logs vault-proxy`) and the persistent JSONL file (`/var/log/vault-proxy/requests.jsonl` on named volume `opentrapp_vault-proxy-logs`) contained tokens in cleartext. Additionally, mitmproxy's built-in per-flow summary printer emitted the same URLs on stdout, bypassing our addon's log path entirely.

**Exposure scope:** An attacker holding the leaked token could authenticate as `@LogoTrappBot` to Telegram — receive/send messages as NewLogoTrappBot. Could NOT cross into the user layer (pairing is gated by a separate handshake that wasn't in the log stream). No user-message content, no pairing codes, no conversations were logged.

**Detection:** During Phase 0 manual smoke-testing, before the automated harness was built.

**Fix** (submodule commit `4f5b560`, parent commit `0ac3e9e`):
- New `BOT_TOKEN_PATH_RE` regex + `_redact_url()` helper in `vault-proxy.py`
- Applied at all 6 `_log_event()` call sites
- `running()` hook sets `ctx.options.flow_detail = 0` to silence mitmproxy's built-in flow summary
- Old token revoked via BotFather + rotated; old requests.jsonl volume purged
- Verified: `grep -c "$TELEGRAM_BOT_TOKEN"` against post-fix `podman logs vault-proxy` returns 0 under live traffic

**Severity before fix:** HIGH (a leaked token on disk compromises the bot identity of every current and future user of this stack).
**Severity after fix:** N/A — fixed.

### Finding #2 — /proc/mounts discloses host block device (`/dev/sda2`)

**Category:** Information disclosure (NOT host reachability).
**Surface:** `podman exec vault-agent cat /proc/mounts` includes lines like:
```
/dev/sda2 /opt/proxy-ca ext4 ro,nosuid,nodev,noatime 0 0
/dev/sda2 /home/vault/.openclaw ext4 rw,nosuid,nodev,noatime 0 0
/dev/sda2 /home/vault/workspace/skills ext4 ro,nosuid,nodev,noatime 0 0
```
The container doesn't have access to `/dev/sda2` as a block device — it can only access the named volumes mounted on top. But the device name leaks, telling an agent/attacker what the host's root partition is.

**Exposure scope:** Low. An attacker already has to be inside vault-agent (meaning they're past container isolation, seccomp, cap_drop, read-only root). The leaked info is the host's disk layout — useful for reconnaissance if combined with a container-escape 0-day, not useful alone.

**Status:** Not fixed; flagged for future hardening. Options: mask `/proc/mounts` with a tmpfs overlay, or switch to rootless overlay storage that obscures the device name.

**Severity:** LOW.

The Telegram-driven tests (`test_smoke.py`, `test_baseline.py`, …, `test_dynamic_shell.py`) are written and ready but need one human action: pasting the Telegram login code on first Telethon run. See `tests/e2e-telegram/FIRST_RUN.md` for the 3 commands to execute.

## Verdict

On the **host-reachability axis** — "can the containerized agent reach the user's machine?" — the answer is **no**. 22 direct probes confirm it.

On the **observability-layer axis** — "does the security infrastructure leak secrets to its own logs?" — there was **one real finding** (Finding #1), now fixed.

On the **prompt-injection axis** — "can the LLM be coerced into exfiltrating or escaping?" — **pending Phase 3+4 execution.**

The "just right" claim at the intro is on the first two axes only. We haven't yet stress-tested the LLM itself. That's tomorrow morning's run.
