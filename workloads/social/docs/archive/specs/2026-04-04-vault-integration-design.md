# Spec: Pioneer → Vault Feed Scanning Integration

**Date:** 2026-04-04
**Phase:** Pioneer Phase 4
**Status:** Design complete, implementation deferred until Moltbook domains enter allowlist
**Cross-reference:** `opencli-container/docs/specs/2026-03-30-feed-scanning-deferred.md` (vault-side deferral decision)

## Problem

When Moltbook domains are added to the vault proxy allowlist, the agent will receive social feed content containing potential prompt injection attacks. Pioneer has 25 injection patterns tuned for this content. The vault proxy needs to inspect Moltbook API responses and block critical-severity injections before the agent sees them.

## Architecture

```
Agent requests Moltbook API
    ↓
vault-proxy.py intercepts response
    ↓
Is domain a Moltbook domain?
    ├── No → pass through (existing behavior)
    └── Yes → parse JSON body
                ↓
          Extract post/comment content
                ↓
          Run 25 patterns against content
                ↓
          ├── Critical match → block response, return sanitized version, log finding
          ├── High/Medium match → pass through, log finding
          └── No match → pass through
```

Scanning happens at the **proxy level (host-side, trusted)**. The agent never sees critical-severity content. This maintains the zero-trust architecture: the agent cannot influence or bypass the scanning.

## Pattern Export Format

Pioneer generates a stripped YAML export via `make export-patterns`:

```yaml
# Generated by: make export-patterns
# Source: config/injection-patterns.yml
# Count: 25 patterns
patterns:
  - id: auth-001
    severity: HIGH
    regex: "(?i)(as the|i am the|this is the).{0,20}(admin|moderator|system|official|moltbook team)"
  - id: inj-001
    severity: CRITICAL
    regex: "(?i)ignore (all |your )?(previous|prior|above) (instructions|prompts|rules)"
  # ... (all 25 patterns)
```

**Format decisions:**
- YAML (not JSON) — matches the source format, human-readable, easy to diff
- `(?i)` flags preserved — Python's `re.compile()` handles them natively. The strip-for-grep workaround in the bash scanner is not needed in Python.
- Only `id`, `severity`, and `regex` fields — category and description are for human display, not proxy logic
- Generated file lives at `data/patterns-export.yml` (gitignored, generated on demand)

**Export script:** A `scripts/export-patterns.py` (Python, since the proxy is Python) that:
1. Reads `config/injection-patterns.yml`
2. Validates all regexes compile with `re.compile()`
3. Writes `data/patterns-export.yml`
4. Reports pattern count and any compile failures

## Proxy Integration

### Loading Patterns

`vault-proxy.py` loads patterns at startup from a configurable path:

```python
MOLTBOOK_PATTERNS_FILE = os.environ.get("MOLTBOOK_PATTERNS", "")

def load_moltbook_patterns():
    """Load pioneer's injection patterns for feed scanning."""
    if not MOLTBOOK_PATTERNS_FILE or not os.path.exists(MOLTBOOK_PATTERNS_FILE):
        return []
    with open(MOLTBOOK_PATTERNS_FILE) as f:
        data = yaml.safe_load(f)
    patterns = []
    for p in data.get("patterns", []):
        try:
            compiled = re.compile(p["regex"])
            patterns.append({"id": p["id"], "severity": p["severity"], "regex": compiled})
        except re.error:
            pass  # skip invalid patterns, log warning
    return patterns
```

Patterns are loaded once at startup. No hot-reload needed — proxy restart picks up changes.

### Response Inspection

In the existing `response()` hook, after the domain allowlist check:

```python
MOLTBOOK_DOMAINS = {"moltbook.com", "api.moltbook.com"}

def scan_moltbook_response(content_text, patterns):
    """Scan response content against injection patterns. Returns list of findings."""
    findings = []
    for p in patterns:
        if p["regex"].search(content_text):
            findings.append({"id": p["id"], "severity": p["severity"]})
    return findings
```

### Blocking Policy

| Severity | Action | Agent sees content? |
|----------|--------|---------------------|
| CRITICAL | Block response, return `{"error": "content_blocked", "reason": "injection_pattern_detected"}` | No |
| HIGH | Log finding to `requests.jsonl`, pass response through | Yes |
| MEDIUM | Log finding to `requests.jsonl`, pass response through | Yes |

The sanitized response uses HTTP 200 with a JSON body: `{"posts": [], "feed_scan": "content_filtered"}`. The agent sees an empty result, not an error — it simply finds no posts matching its query, which is the safe outcome. The `feed_scan` key signals to any monitoring that filtering occurred.

### Logging

Findings are logged to the existing `requests.jsonl` with a new `feed_scan` field:

```json
{
  "timestamp": "2026-04-15T10:30:00Z",
  "method": "GET",
  "url": "https://api.moltbook.com/posts?limit=50",
  "status": 200,
  "feed_scan": {
    "posts_scanned": 50,
    "findings": [
      {"id": "inj-001", "severity": "CRITICAL", "post_index": 3},
      {"id": "soc-001", "severity": "HIGH", "post_index": 12}
    ],
    "blocked": true,
    "reason": "critical_finding"
  }
}
```

## Activation

Feed scanning activates when ALL of these are true:
1. Moltbook domains are in `proxy/allowlist.txt`
2. `MOLTBOOK_PATTERNS` environment variable points to a valid patterns file
3. Proxy is restarted after both conditions are met

If either condition is missing, scanning is dormant — responses pass through unmodified.

2. Moltbook response inspection in `response()` hook
3. Blocking logic for critical findings
4. Logging of findings to `requests.jsonl`
5. `MOLTBOOK_PATTERNS` env var in `compose.yml`
6. Verification: add feed scan check to `verify.sh`

## Testing Strategy

**Pioneer-side (offline):**
- Export script produces valid YAML from `injection-patterns.yml`
- All 25 regexes compile with Python `re.compile()`
- Export file contains exactly 25 entries
- Known malicious content matches expected patterns in Python

**Vault-side (container):**
- With patterns loaded and Moltbook domain in allowlist:
  - Malicious fixture response → blocked, logged
  - Clean fixture response → passed through, no findings logged
  - Mixed response → critical posts blocked, clean posts visible
- Without patterns loaded: all responses pass through unmodified

## Security Considerations

- Pattern file is read-only to the agent (lives on host side, loaded by proxy)
- Agent cannot influence which patterns are loaded or how they're applied
- Blocking replaces the entire response, not individual posts — prevents the agent from inferring which specific content was flagged
- False positives on CRITICAL patterns will block legitimate content. Patterns must be conservative. The existing 8 CRITICAL patterns target obvious injection (e.g., "ignore all previous instructions") with low false-positive risk.
